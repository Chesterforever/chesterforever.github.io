TI AM243x-LP USB驱动开发记录

TI AM243x-LP USB驱动开发寄录

不得不承认之前被ST的软件生态养的太好了，第一次用TI的Soc开发板，踩了很多坑，但是狠狠恶补了一波多核和Soc的知识点，不亏。

相关资料

TI官网上给开发者提供的相关资料有

  • AM243x芯片手册
  • AM243x-LP开发板硬件手册
  • SDK（最新版本SDK分为mcu+sdk，motor control sdk，industrial sdk三种）
  • SDK使用手册

SDK使用手册直接用浏览器打开然后用浏览器的翻译功能就行,芯片手册和开发板硬件手册只有英文版pdf，所以我用了Zotero+pdf2zh插件进行翻译（这个是github开源的pdf翻译方案，而且有双语对照功能，还是挺好用的）

我想把翻译好的芯片手册和开发板硬件手册的链接放在下面，但是发现github传不了100Mb以上的文件，等我有空研究一下

踩坑过程

BOOT过程

因为网上关于TI开发板的视频资料较少，所以基本上是直接根据SDK使用手册一步一步做的，这就导致了一旦使用手册里有写的不明确的地方，就要在这些地方卡一段时间去试错。

我遇到的第一个问题是芯片如何正确启动并初始化。就像其他的高级Soc一样，TI为AM2434的启动设置了多阶段引导加载程序，也就是多级bootloader。

首先是第一级bootloader，出于安全考虑，TI在芯片出厂前将其保存在安全的只读内存ROM中，称为ROM引导加载程序（RBL）。RBL程序负责引导进入下一级bootloader，而不需要知道芯片的其他内核。

然后是第二级bootloader，主要负责处理不同内核上的复杂引导加载，称为SBL（second bootloader）。加载多核应用程序比加载单核应用程序要复杂很多，要考虑映像准备和共享内存访问等问题。好在TI提供了相关的makefile文件makefile_ccs_bootimage_gen，方便用户自定义编译出需要的SBL文件。

在了解完这些前置知识点后，就可以老老实实一步一步跟着SDK使用手册的步骤走了

这个时候我才知道原来TI的这款芯片还分了三种安全等级，分别对应

  • GG 非安全 适用固件后缀为无后缀
  • HS 高安全 适用固件后缀为_hs
  • FS 现场安全 适用固件后缀为_fs

除了看芯片丝印信息这一方法来得知安全等级信息，在逛TI论坛时我还发现了一个方法来看自己的芯片是GP还是HS_FS，这里就不多介绍了，放个链接[FAQ] [参考译文] [常见问题解答] TDA4VM：如何检查器件类型(GP FS)？ - 处理器（参考译文帖）(Read Only) - 处理器（参考译文帖） - E2E™ 设计支持

但其实从22年9月后TI就逐渐取消了对GP型号的生产和支持，SDK在9.0版本后就完全不支持GP型号了，所以基本上现在市面上买到的器件都是HS_FS型号的，所以不用担心这个问题，直接用最新版的SDK就行。

而我的芯片由于是GP型号的，只能去官网上下载了08030018版本的SDK和1.12.1版本的sysconfig工具，并导入到CCS的product中。如果是HS_FS型号的开发板，版本下载官网最新的就行。

在经历了这些后，我终于在老版本的SDK上找到了后缀不带.hs_fs的SBL文件并成功复现了SDK使用手册中的步骤。这个时候就可以将LP启动模式切换到OSPI模式了，然后跟着使用手册的步骤应该没有什么坑了，遇到问题就问AI，大概率能解决。执行完OSPI boot之后，就可以直接用CCS直接烧录和调试程序了，不需要每次切换回UART boot模式。

在终端执行命令行时尽量用相对路径而不用绝对路径，这样能有效避免应命令引用路径过长导致的问题

如果是HS_FS型号的开发板，要装OpenSSL，当编译好一个程序镜像后，TI 的构建流程会调用 OpenSSL 来生成一个数字签名（就像给软件盖一个加密的印章）。芯片上的 ROM 引导程序或二级引导程序在加载应用前，会使用预先烧录在芯片中的公钥来验证这个签名。如果验证失败，说明软件可能被篡改，芯片将拒绝执行，从而防止恶意代码运行

XIP Image 的全称是 eXecute-In-Place Image，中文可译为 “就地执行映像” 或 “直接执行映像”。

它是一种存储在非易失性存储器（如 Flash）中，且能够被 CPU 直接从中读取并执行的程序二进制文件，而无需将其先复制到 RAM（内存）中，这个是由链接脚本决定的。

用于快速启动，节省RAM空间，但是执行速度较慢，写入困难

USB CDC，USB HID，tiny USB相关知识点

1. USB CDC (Communications Device Class)
  • 是什么？ 一种 USB 设备类，专门用于通信设备。
  • 目的：让 USB 设备能够被主机识别为一个传统的通信设备，最典型的就是虚拟串口 (VCP - Virtual COM Port)。
  • 工作原理：设备通过 USB 总线模拟出一个串口的所有功能。主机操作系统加载标准的CDC驱动程序，创建一个 COM 端口（Windows）或 tty 设备（Linux/macOS）。应用程序（如串口助手、终端）像操作普通串口一样与之通信，数据实际上通过 USB 传输。
  • 特点：需要驱动，但现代操作系统通常内置了标准 CDC 驱动（USB Serial Driver）。传输带宽高，适合数据量较大的通信（如日志输出、文件传输）。延迟相对较高，不如 HID 实时。
  • 典型应用：单片机调试日志输出、通过 AT 指令控制通信模块（4G、Wi-Fi）、设备固件升级（DFU）、与 PC 进行自定义数据通信。
2. USB HID (Human Interface Device)
  • 是什么？ 一种 USB 设备类，专门用于人机交互设备。
  • 目的：让 USB 设备能够与主机进行简单、标准化的交互，主要用于输入设备。
  • 工作原理：设备必须严格按照 HID 类规范定义其报告描述符，向主机报告它是什么设备（键盘、鼠标、游戏手柄等）以及能发送哪些数据（按键、坐标、滚轮等）。数据通过报告的形式进行传输。
  • 特点：免驱！这是最大的优点。操作系统都内置了标准的 HID 驱动，设备插入后即可立即使用。传输带宽低，延迟低，实时性好。因为数据量小（一次按键才几个字节）。配置复杂，需要正确定义报告描述符。
  • 典型应用：自定义键盘、鼠标、游戏手柄、旋钮控制器、传感器数据上报（但数据量不能大）。
3. TinyUSB
  • 是什么？ 一个开源的、跨平台的、用于嵌入式系统的 USB 设备协议栈。
  • 目的：为资源受限的 MCU 提供一个轻量级、功能丰富、易于移植的软件库，帮助开发者更容易地实现各种 USB 功能。
  • 工作原理：它实现了 USB 底层协议，并提供了对多种 USB 设备类（Class）的支持，包括：CDC（实现虚拟串口）HID（实现键盘、鼠标等）MSC（实现U盘）MIDI等等…
  • 特点：不是协议标准，而是实现工具。轻量级，适合 RAM 和 Flash 有限的单片机。可移植性强，支持众多芯片架构（ARM Cortex-M, RISC-V, ESP32-Sx等）和厂商（ST, NXP, Microchip, Raspberry Pi等）。简化开发，你不需要深入理解复杂的 USB 底层协议，只需调用 TinyUSB 提供的 API 并配置你需要的功能即可。

需要注意的是TI移植的tiny USB库只支持CDC设备类

将USB CDC示例代码从R5F内核移植到M4F内核上

这里需要考虑的地方还是挺多的，真的非常烧脑。首先查看Soc技术手册的第二章Memory map会发现，Soc的设备域被分为了main域和mcu域，其中4个R5F内核被放在了main域中，M4F内核被放在了mcu域中，而USB外设被放在了main域中。所以我们不能简单地用mcu域中的M4F去访问main域的USB外设，而是得通过域间互联总线进行跨域访问比如L3或CPPI总线。并且，技术手册的2.3节和2.4节还提到了有些处理器仅支持32位寻址，无法访问较高地址范围，所以得使用RAT模块将32位地址重映射为36位或48位地址来访问较高地址范围。然后6.3.3.7.1节里详细解释了M4F为什么需要RAT，以及给出了这样一句话“M4F可以使用地址0xA0000000-0xDFFFFFFF来访问Soc级别的各种内存端点”，所以按照我的理解，只要把USB和dma操作相关的内存缓冲区设置为Soc级别的内存，就可以顺利完成移植。

好在TI的syscfg工具为RAT提供了图形化配置界面，这样就不用去看RAT的寄存器操作了

这里必须吐槽一下，技术手册里提供了推荐的RAT配置，按照手册的说法，M4F内核使用它的配置可以实现访问Soc的所有外设。但是！这个配置真的不太敢用，因为这个技术手册虽然写了支持AM64x/AM243x两种型号，但是根据整个手册的写法来看，这个配置很有可能只适用于AM64x

比较难受的是TI的SDK不支持M4F内核的USB，所以只能手动导入相关的tiny USB库并完成USB驱动的配置，然后还要修改ld链接文件和makefile。因为目前涉及到跨域间的通信，所以M4F的MPU配置也需要配合链接文件修改。

从示例代码中可以看出，在链接文件中对于内存的划分是以自身内核视图而非全局视图进行的，也就是肯定会从0x00000000开始划分，第一个是复位向量表区域，第二个是IRAM区域……

结合技术手册2.3节内存映射视图来看会发现链接文件只用了RAM0和DRAM，RAT_REGION0三个区域去定义内存，那为什么其他区域在程序中也用到了但是却不需要链接文件中定义呢？

这是因为链接脚本主要目的是控制程序段的分配，通常只定义那些需要直接存储程序内容的内存区域。这些其他区域都是Coresight调试组件或系统控制寄存器，由调试器或系统软件直接访问，而不需要链接文件分配内容。
